# コードレビューのコメントの書き方



## まとめ

-   親切であること。
-   理由を説明すること。
-   明確な指示を与えることと、問題を指摘するだけにとどめて作成者に判断を委ねることのバランスを取ること。
-   複雑さをレビュアーに説明するのではなく、コードを単純化するかコード内にコメントを追加するよう作成者に促すこと。

## 礼儀

一般に、レビュー対象のコードを書いた作成者に対して[礼儀正しく敬意を払う](https://chromium.googlesource.com/chromium/src/+/master/docs/cr_respect.md)ことは重要であり、同時に非常に明確で役立つコメントを心がける。
その一つの方法は、コメントは常に*コード*について述べ、*作成者*については決して述べないようにすることだ。
この方法を常に守る必要はないが、相手を動揺させたり対立を招いたりしかねないことを言うときには、必ず使うべきだ。
たとえば次のようになる。

悪い例: 「ここで並行性から得られるメリットが明らかにないのに、なぜ**あなた**はスレッドを使ったのですか?」

良い例: 「ここでの並行性モデルは、見たところ実際のパフォーマンス上のメリットがないままシステムに複雑さを加えています。パフォーマンス上のメリットがないので、このコードは複数スレッドを使うよりシングルスレッドにするのが一番です。」

## 理由を説明する {#why}

上の「良い例」で気づくのは、それがなぜそのコメントをするのかを作成者が理解する助けになっている点だ。
レビューコメントにこの情報を必ず含める必要はないが、自分の意図、従っているベストプラクティス、その提案がどうコードの健全性を改善するのかについて、もう少し説明を添えるのが適切なこともある。

## 指針を与える {#guidance}

**一般に、CLを直すのはレビュアーではなく作成者の責任である。**
解決策の詳細な設計をしたり、作成者のためにコードを書いたりする義務はない。

とはいえ、これはレビュアーが非協力的でよいという意味ではない。
一般に、問題を指摘することと直接的な指針を与えることの間で適切なバランスを取るべきだ。
問題を指摘して作成者に判断を委ねることは、作成者の学びにつながることが多く、コードレビューもしやすくなる。
また、レビュアーよりも作成者のほうがコードに近いため、より良い解決策につながることもある。

ただし、直接的な指示、提案、あるいはコードそのもののほうが役立つこともある。
コードレビューの第一の目標は、可能な限り良いCLを得ることだ。
第二の目標は、開発者のスキルを高め、時間とともに必要なレビューがどんどん少なくなるようにすることである。

人は、改善できる点だけでなく、うまくできている点を補強されることからも学ぶことを忘れないでほしい。
CLの中に気に入った点があれば、それについてもコメントしよう。
例: 作成者が雑然としたアルゴリズムを整理した、模範的なテストカバレッジを追加した、あるいはレビュアーであるあなたがそのCLから何かを学んだ、など。
すべてのコメントと同様に、なぜそれを気に入ったのか[理由](#why)を含め、良い習慣を続けるよう作成者をさらに後押しする。

## コメントの重要度を示す {#label-comment-severity}

コメントの重要度を示し、必須の変更とガイドラインや提案とを区別することを検討する。

例を挙げる。

> Nit: これは些細なことです。厳密には対応すべきですが、大きな影響はありません。
>
> Optional（または Consider）: これは良い考えだと思いますが、必須ではありません。
>
> FYI: このCLで対応してほしいわけではありませんが、今後のために考えておくと面白いかもしれません。

これにより、レビューの意図が明確になり、作成者がさまざまなコメントの重要度を判断しやすくなる。
誤解を避けるのにも役立つ。
たとえば、コメントにラベルがないと、作成者はすべてのコメントを必須と解釈してしまうことがある。
一部のコメントが単に情報提供や任意の意図であっても、である。

## 説明を受け入れる {#explanations}

理解できないコードについて作成者に説明を求めると、たいていは**コードをより明確に書き直す**結果になるべきだ。
ときには、過度に複雑なコードをただ説明するだけでない限り、コード内にコメントを追加するのも適切な対応となる。

**コードレビューツール上だけに書かれた説明は、将来コードを読む人の役には立たない。**
それが許容されるのは、たとえば自分があまり詳しくない領域をレビューしていて、そのコードを普通に読む人なら既に知っているはずのことを作成者が説明する場合など、ごく一部の状況に限られる。
