# コードレビューの基準



コードレビューの主目的は、Googleのコードベース全体のコードの健全性が時間とともに向上していくことを保証することである。コードレビューのツールやプロセスはすべて、この目的のために設計されている。

これを達成するためには、一連のトレードオフをバランスさせる必要がある。

第一に、開発者は自分のタスクを_前進させられる_必要がある。コードベースへの改善を一度もサブミットしなければ、コードベースは決して改善されない。また、レビュアーが_いかなる_変更も入りにくくしてしまうと、開発者は将来改善を行う意欲を失う。

一方で、各CLが、時間が経ってもコードベース全体のコードの健全性を低下させない程度の品質であることを保証するのは、レビュアーの責務である。これは難しいことがある。なぜなら、コードベースはしばしば、時間をかけてコードの健全性が少しずつ低下することで劣化していくからである。とくにチームが大きな時間的制約の下にあり、目標達成のために近道を取らざるを得ないと感じているときにそうなる。

また、レビュアーはレビューしているコードに対するオーナーシップと責任を持つ。レビュアーは、コードベースが一貫性、保守性、その他[「コードレビューで何を見るか」](looking-for.md)で述べたすべての性質を保つことを確実にしたいと考える。

こうして、コードレビューで期待される基準として次の規則が得られる。

**一般に、レビュアーは、CLが取り組んでいるシステム全体のコードの健全性を確実に向上させる状態に達したら、たとえそのCLが完璧でなくても、承認する方向に傾けるべきである。**

これがコードレビューのガイドライン全体のなかで最上位の原則である。

もちろん、これには限界がある。たとえば、CLがレビュアーのシステムに望ましくない機能を追加するものであれば、コードがよく設計されていても、レビュアーは承認を拒否してよい。

ここで重要な点は、「完璧な」コードなど存在しないということである。あるのは_より良い_コードだけである。レビュアーは、承認を与える前に、CLのあらゆる細部を磨き上げるよう作成者に要求すべきではない。むしろレビュアーは、前進を続ける必要性と、自分が提案している変更の重要性とを比較してバランスを取るべきである。完璧を求めるのではなく、レビュアーが求めるべきは_継続的な改善_である。全体として、システムの保守性、可読性、理解しやすさを向上させるCLを、「完璧」でないからといって何日も何週間も遅らせるべきではない。

レビュアーは、何かをもっと良くできるという意見を述べるコメントを_いつでも_自由に残してよい。ただし、それがあまり重要でないなら、「Nit: 」のような接頭辞を付けて、それが単に磨き上げの点であり作成者が無視してもよいことを伝えること。

注意: このドキュメントのいかなる内容も、システム全体のコードの健全性を確実に_悪化させる_CLをチェックインすることを正当化しない。それを行うのは[緊急事態](../emergencies.md)のときだけである。

## メンタリング

コードレビューには、言語、フレームワーク、あるいは一般的なソフトウェア設計の原則について、開発者に何か新しいことを教えるという重要な機能がありうる。開発者が何か新しいことを学ぶのを助けるコメントを残すのは、いつでも構わない。知識の共有は、時間をかけてシステムのコードの健全性を向上させることの一部である。ただし、コメントが純粋に教育的なもので、このドキュメントで述べた基準を満たすうえで決定的でない場合は、「Nit: 」を付けるか、それがこのCLで作成者が必ず解決しなければならないものではないことを示すこと。

## 原則 {#principles}

*   技術的な事実とデータは、意見や個人的な好みに優先する。

*   スタイルの問題については、[スタイルガイド](http://google.github.io/styleguide/)が絶対的な権威である。スタイルガイドにない純粋なスタイル上の点（空白など）は個人的な好みの問題である。スタイルは既存のものと一貫させるべきである。既存のスタイルがない場合は、作成者のものを受け入れること。

*   **ソフトウェア設計の側面は、純粋なスタイルの問題や単なる個人的な好みであることはほとんどない。** それらは根底にある原則に基づいており、単なる個人の意見ではなく、それらの原則に照らして判断すべきである。場合によっては、いくつかの妥当な選択肢があることもある。作成者が（データによってか、あるいは堅実なエンジニアリングの原則に基づいて）複数の手法が同等に妥当であることを示せるなら、レビュアーは作成者の好みを受け入れるべきである。そうでなければ、選択はソフトウェア設計の標準的な原則によって決まる。

*   他にどの規則も当てはまらない場合、システム全体のコードの健全性を悪化させない限り、レビュアーは作成者に現在のコードベースにあるものとの一貫性を保つよう求めてよい。

## 対立の解消 {#conflicts}

コードレビューで対立が生じた場合、最初のステップは常に、開発者とレビュアーが、このドキュメントや[CL作成者ガイド](../developer/index.md)およびこの[レビュアーガイド](index.md)にある他のドキュメントの内容に基づいて、合意に達しようとすることであるべきだ。

合意に達するのがとくに困難になった場合は、コードレビューのコメントだけで対立を解決しようとするのではなく、レビュアーと作成者の間で対面の会議やビデオ会議を持つと役立つことがある。（ただしそうする場合は、将来の読み手のために、議論の結果をCLのコメントとして必ず記録すること。）

それでも状況が解決しない場合、最も一般的な解決方法はエスカレーションである。エスカレーションの経路としては、より広いチームでの議論、テックリードに意見を求める、コードのメンテナーに判断を求める、エンジニアリングマネージャーに助けを求める、といったものがよくある。**作成者とレビュアーが合意に達せないからといって、CLを放置してはならない。**
