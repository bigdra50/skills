代替案

案A: IScenarioTitleProvider 抽象化 採用

Application 層に title 解決の interface を導入。locale を内包。

メリット
  • 呼び出し側が locale を知らなくて良い
  • テストが容易 (mock 注入)
  • fallback ロジックを 1 箇所に集約
デメリット
  • interface が増える
  • DI 登録の手間
案B: extension method 維持 却下

現状の ScenarioMetadataExtensions を残す。

メリット
  • 変更コストゼロ
  • 追加の抽象化なし
デメリット
  • 呼び出し側が locale を引き回す (情報漏洩)
  • fallback の重複実装リスク
案C: ScenarioMetadata に title 解決を移動 却下

Domain record に locale 対応 method を生やす。

メリット
  • 呼び出し側がシンプル
デメリット
  • Domain 層に locale 概念が漏れる (層越境)
  • 純粋関数性が壊れる
  • CLAUDE.md NRT ポリシーと整合しづらい

決定

採用案: 案A (IScenarioTitleProvider 抽象化)
決定マトリクスで最高得点 (38)。特に「層分離の整合性」と「テスト容易性」で優位。

採用理由

  • 呼び出し 4 箇所から locale 知識を除去でき、情報漏洩を解消する
  • Application 層に置くため、依存方向ルールに違反しない
  • fallback ロジックを将来拡張する余地がある

結果として生じる変更

  • Application/Localization/IScenarioTitleProvider.cs 新規追加
  • Infrastructure/Localization/LocaleAwareScenarioTitleProvider.cs 新規追加
  • 4 つの ScreenFlow で ResolveTitle(id, locale) → ResolveTitle(id) に置換
  • ScenarioMetadataExtensions.cs 削除
