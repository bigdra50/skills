# レビュー中のCLをたどる



## まとめ

[何を見るべきか](looking-for.md)が分かったところで、複数ファイルにまたがるレビューを最も効率よく進める方法を考える。

1.  その変更は理にかなっているか。良い[説明](../developer/cl-descriptions.md)が付いているか。
2.  変更の最も重要な部分を最初に見る。全体としてうまく設計されているか。
3.  CL（チェンジリスト）の残りの部分を適切な順序で見ていく。

## ステップ1: 変更を俯瞰する {#step_one}

[CLの説明](../developer/cl-descriptions.md)と、そのCLが全体として何をするのかを見る。
この変更はそもそも理にかなっているか。
この変更が本来行われるべきでなかったものなら、なぜその変更を行うべきでないのかを説明し、ただちに返信する。
このように変更を却下するときは、作成者が代わりに何をすべきだったかを提案するのもよい。

たとえば、次のように伝えるとよい。
「しっかり取り組んでもらえたようで、ありがとうございます。
ただ、ここで修正している FooWidget システムは、実は削除する方向に進めているところで、今は新たな修正を加えたくありません。
代わりに、新しい BarWidget クラスのリファクタリングをしてもらえませんか。」

ここで注目したいのは、レビュアーが今のCLを却下して代替案を提示しただけでなく、それを*丁寧に*行った点である。
この種の丁寧さは重要だ。
意見が食い違うときでも、開発者として互いを尊重していることを示したいからである。

行いたくない変更を含むCLが何件も続くようなら、CLが書かれる前にもっとコミュニケーションが取られるよう、チームの開発プロセスや外部コントリビューター向けに公開しているプロセスを見直すことを検討すべきだ。
大量の作業をして、それを破棄したり大幅に書き直したりしなければならなくなる前に、「ノー」と伝えるほうがよい。

## ステップ2: CLの主要部分を確認する {#step_two}

このCLの「主要」部分にあたるファイルを見つける。
たいていの場合、論理的な変更が最も多いファイルが1つあり、それがCLの中心となる部分だ。
まずこの主要部分を見る。
これにより、CLの細かい部分すべてに文脈が与えられ、コードレビューが全体として速くなる。
CLが大きすぎてどこが主要部分か分からない場合は、まず何を見ればよいか作成者に尋ねるか、[CLを複数のCLに分割する](../developer/small-cls.md)よう依頼する。

この部分に大きな設計上の問題が見つかった場合は、今は残りのCLをレビューする時間がなくても、そのコメントをただちに送るべきだ。
実際、残りのCLをレビューするのは時間の無駄になりかねない。
設計上の問題が十分に大きければ、レビュー対象の他のコードの多くは消えてなくなり、結局問題でなくなるからである。

こうした主要な設計に関するコメントをただちに送ることが重要な理由は、主に2つある。

-   作成者はしばしば、CLを送ったらレビューを待つ間にそのCLを土台にしてすぐ次の作業を始める。
    レビュー中のCLに大きな設計上の問題があれば、後続のCLも作り直さなければならなくなる。
    問題のある設計の上に余計な作業を重ねすぎる前に、作成者に気づかせたい。
-   大きな設計変更は、小さな変更よりも時間がかかる。
    開発者のほぼ全員が締め切りを抱えている。
    その締め切りを守りつつコードベースの品質を保つには、CLの大きな作り直しにできるだけ早く着手する必要がある。

## ステップ3: CLの残りを適切な順序で見ていく {#step_three}

CL全体に大きな設計上の問題がないことを確認したら、どのファイルも見落とさないようにしつつ、ファイルを見ていく論理的な順序を考える。
通常、主要なファイルを見終わったあとは、コードレビューツールが提示する順番どおりに各ファイルを見ていくのが最も簡単だ。
本体のコードを読む前にテストを先に読むと、その変更が何をしようとしているのかが分かるので、役立つこともある。
