# PR Brief

ブランチ間差分からレビュワーが通読できるブリーフィング資料を生成する。

## 引数

```
pr-brief [<base>] [--out <path>]
```

- `<base>`: 比較ベース。省略時 `main`
- `--out`: 出力先。省略時 `docs/reviews/<branch-name>.md`

## Workflow

### 1. 差分情報の収集

並行で実行:

```bash
git diff --stat <base>...HEAD    # 変更ファイルと行数
git log <base>..HEAD --oneline   # コミット履歴 (意図の手がかり)
git diff <base>...HEAD           # 変更内容本体
git status --short               # 未コミット変更の有無
```

未コミット変更があれば、含めるかユーザーに確認する。

### 2. 変更の分類

変更ファイルをカテゴリ分けする:

| カテゴリ | 典型パス |
|---|---|
| 新機能 | 新規 `.cs`、新規ディレクトリ |
| ロジック変更 | 既存ランタイムコード |
| テスト | `Tests/` 配下 |
| ドキュメント | `docs/`, `*.md`, `ADR-*.md` |
| アセット/データ | `Resources/`, `.asset`, `.meta`, `.prefab`, `.unity` |
| 設定 | `*.json`, `*.toml`, `ProjectSettings/` |
| スクリプト/ツール | `scripts/`, `mise`, `.editorconfig` |

### 3. レビュー論点の抽出

各カテゴリ・主要ファイルごとに以下の観点で論点を立てる:

- 設計判断 (WHY)
- 影響範囲 (どこが壊れ得るか)
- 仕様整合性 (ADR・既存ドキュメントと照合)
- テストがカバーしない経路
- 性能・セキュリティ・エラー処理・並行性

不明点は `?` で明示し、読者判断に委ねる。

### 4. 雛形に沿って出力

- 雛形: [assets/skeleton.md](assets/skeleton.md)
- 章立て詳細 (PR種別ごと): [references/templates.md](references/templates.md)

雛形を読み込み、プレースホルダ `{...}` を実データで埋める。余計な章は削除、必要なら追加する。

### 5. 後続アクション提案

出力後、ユーザーに以下を提案:

- PR 本文へコピー
- `/claude-review`, `/copilot-review`, `/dual-review` の入力として利用
- チーム読み合わせ

## 注意

- コード全文は貼らない。要点抜粋 + `file:line` 参照に留める
- 意図不明な変更は `?` マーク
- ADR・既存ドキュメントがあれば必ず参照
- 画像/スクリーンショットはプレースホルダで残す (埋めるのは人間)
- 1章あたり30行以内を目安に保ち、通読集中力を保つ

## PR 種別ごとのテンプレート選択

- **機能追加**: アーキテクチャ図・設計判断を厚く
- **リファクタ**: before/after・メトリクス差分を厚く
- **バグ修正**: 再現手順・根本原因・回帰リスクを厚く
- **ドキュメント/設定のみ**: サマリ + 差分要点で十分

詳細な章立ては [references/templates.md](references/templates.md) を参照する。

## 類似スキルとの使い分け

- **diff-review**: 単純に diff を整形表示するだけ。人間がレビューする素材提供
- **claude-review / copilot-review / codex-review**: 自動でコードレビューを実施し指摘を出す
- **pr-brief (本スキル)**: 変更の意味づけ・論点抽出を行い、レビュワー向けの読み物を生成

単なる表示でも自動レビューでもなく、「**レビュー前の事前説明資料**」を作りたいときに使う。
